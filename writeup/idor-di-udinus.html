IDOR - Unauthorized Data Modification

Irsyad Bayu Kurniawan — Web Security Researcher

Target

Method: POST
Endpoint: /cmember/pencari/profil/edit/ID
Host: cc.dinus.ac.id

Ringkasan

Dalam pengujian terhadap sebuah aplikasi web, ditemukan kerentanan yang memungkinkan pengguna untuk mengubah data milik pengguna lain hanya dengan memodifikasi parameter pada request. Kerentanan ini termasuk dalam kategori Insecure Direct Object Reference (IDOR), yang terjadi akibat tidak adanya validasi otorisasi di sisi server.

Tingkat keparahan dari temuan ini dikategorikan sebagai High karena berdampak langsung pada integritas dan keamanan data pengguna.

Komponen Terdampak

Endpoint: Update Data / Profil User

Awal Penemuan

Temuan ini berawal dari eksplorasi sederhana terhadap fitur perubahan data profil. Secara umum, fitur ini terlihat normal seperti aplikasi lainnya, di mana pengguna dapat mengubah informasi miliknya sendiri.

Namun, dalam proses pengujian lebih lanjut, muncul kecurigaan bahwa sistem hanya mengandalkan parameter yang dikirim dari client tanpa validasi lebih lanjut di server. Hal ini menjadi titik awal untuk melakukan pengujian manipulasi request.

Analisis Teknis

Pada saat melakukan perubahan data, request berhasil diintercept menggunakan tools seperti Burp Suite. Dalam request tersebut ditemukan parameter berikut:

idpencari=401403

Parameter ini merepresentasikan ID user yang datanya akan diproses oleh server. Namun yang menjadi masalah adalah tidak adanya validasi apakah ID tersebut benar-benar milik user yang sedang login.

Dengan mengganti nilai parameter menjadi ID user lain:

idpencari=401404

Server tetap memproses request tersebut tanpa penolakan, yang berarti sistem sepenuhnya mempercayai input dari client.

Proof of Concept

Berikut adalah demonstrasi eksploitasi kerentanan:

  • Melakukan perubahan data sebagai user (ID pelaku)
  • Intercept request menggunakan proxy
  • Mengubah parameter idpencari ke ID user lain
  • Mengirim ulang request ke server
  • Server merespon berhasil
  • Data korban berubah sesuai input penyerang

Dampak

  • Perubahan data tanpa izin (Unauthorized Data Modification)
  • Pelanggaran privasi pengguna
  • Potensi pengambilalihan akun (Account Takeover)
  • Kerusakan integritas data sistem
  • Penyalahgunaan fitur oleh pihak tidak bertanggung jawab

Akar Masalah

Akar utama dari kerentanan ini adalah kegagalan dalam menerapkan kontrol otorisasi di sisi server. Sistem tidak melakukan verifikasi apakah user yang melakukan request memiliki hak akses terhadap resource yang diminta.

Selain itu, penggunaan parameter ID yang sepenuhnya berasal dari input user tanpa validasi memperbesar risiko eksploitasi.

Insight & Pembelajaran

Dari kasus ini, terdapat beberapa pelajaran penting dalam dunia keamanan aplikasi:

  • Validasi di sisi client tidak pernah cukup — semua harus divalidasi ulang di server
  • Setiap request sensitif wajib memiliki pengecekan otorisasi
  • ID yang terlihat “sepele” sering menjadi titik masuk serangan
  • Tools sederhana seperti proxy sudah cukup untuk menemukan celah serius

Kasus ini juga menunjukkan bahwa kerentanan tidak selalu berasal dari sistem yang kompleks. Sering kali, kesalahan sederhana dalam desain otorisasi justru menjadi celah terbesar.

Rekomendasi

  • Validasi bahwa user hanya dapat mengakses datanya sendiri
  • Gunakan session-based authorization
  • Jangan percaya input ID dari client
  • Tambahkan logging aktivitas mencurigakan
  • Lakukan audit keamanan secara berkala

Kesimpulan

Kerentanan ini memungkinkan penyerang untuk memodifikasi data pengguna lain tanpa izin hanya dengan mengubah parameter pada request.

Perbaikan segera diperlukan untuk mencegah penyalahgunaan lebih lanjut dan menjaga integritas sistem serta keamanan data pengguna.
