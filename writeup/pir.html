No Rate Limiting pada Endpoint Registrasi

Irsyad Bayu Kurniawan — Bug Hunter

Target

https://asm.seclab.id/api/auth/register

Latar Belakang

Dalam proses pengujian terhadap fitur registrasi pada aplikasi ASM Seclab, dilakukan analisis terhadap bagaimana sistem menangani request pendaftaran akun baru.

Fitur registrasi merupakan salah satu komponen penting dalam aplikasi berbasis user, karena menjadi entry point utama bagi pengguna untuk mengakses sistem.

Identifikasi Kerentanan

Saat melakukan pengujian melalui antarmuka web (UI), ditemukan adanya validasi tertentu yang membatasi input pengguna. Namun, validasi tersebut ternyata hanya diterapkan pada sisi client.

Ketika request dikirim langsung ke endpoint API menggunakan metode manual, sistem tetap menerima dan memproses data tanpa melakukan validasi tambahan.

Hal ini menunjukkan bahwa validasi pada sisi server tidak diimplementasikan dengan baik.

Analisis Teknis

Endpoint registrasi dapat menerima request berulang dalam jumlah besar tanpa adanya mekanisme pembatasan. Tidak ditemukan adanya:

  • Rate limiting per IP atau per user
  • Delay atau throttling request
  • Validasi CAPTCHA
  • Verifikasi email setelah registrasi

Dengan kondisi ini, attacker dapat dengan mudah mengotomatisasi pembuatan akun menggunakan script sederhana.

Contoh Request

POST /api/auth/register HTTP/1.1 Host: asm.seclab.id Content-Type: application/json { "company_name":"test", "email":"random@test.com", "website":"test.com", "phone":"+6281234567890", "country":"INDONESIA" }

Hasil Pengujian

Request dapat dikirim secara berulang dengan variasi email yang berbeda, dan seluruh request diproses dengan sukses oleh server.

{ "status": true, "message": "Registration successful" }

Tidak ditemukan batasan jumlah request maupun mekanisme proteksi terhadap abuse.

Dampak

  • Pembuatan akun palsu secara massal
  • Pemborosan resource server dan database
  • Penyalahgunaan fitur berbasis akun (trial, kredit, dsb)
  • Spam terhadap sistem internal
  • Penurunan kualitas dan integritas data
  • Potensi abuse lanjutan terhadap fitur lain

Analisis Risiko

Kerentanan ini termasuk dalam kategori abuse vulnerability, di mana sistem tidak memiliki kontrol terhadap jumlah request yang diterima.

Dalam skenario nyata, attacker dapat memanfaatkan celah ini untuk:

  • Membuat ribuan akun dalam waktu singkat
  • Melakukan brute force terhadap sistem lain yang terhubung
  • Menyalahgunakan resource backend secara signifikan

Rekomendasi

  • Menerapkan rate limiting pada endpoint registrasi
  • Menambahkan CAPTCHA untuk mencegah bot
  • Mewajibkan verifikasi email sebelum aktivasi akun
  • Melakukan validasi input pada sisi server
  • Menerapkan monitoring terhadap aktivitas abnormal

Kesimpulan

Ketiadaan mekanisme pembatasan pada endpoint registrasi membuka peluang besar untuk penyalahgunaan sistem.

Implementasi kontrol sederhana seperti rate limiting dan verifikasi email dapat secara signifikan mengurangi risiko eksploitasi.

Temuan ini disusun sebagai bagian dari responsible disclosure dan tidak disalahgunakan.
