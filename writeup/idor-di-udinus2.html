IDOR - Unauthorized Access to User Data

Irsyad Bayu Kurniawan — Web Security Researcher

Target

Method: POST
Endpoint: /newdaftar/get_all_data
Host: https://apply.dinus.ac.id

Ringkasan

Dalam pengujian lanjutan terhadap sebuah sistem pendaftaran online, ditemukan kembali kerentanan dengan pola yang serupa, yaitu Insecure Direct Object Reference (IDOR). Kerentanan ini memungkinkan pengguna untuk mengakses data pribadi pengguna lain hanya dengan memodifikasi parameter sederhana pada request.

Temuan ini menunjukkan adanya kelemahan fundamental dalam mekanisme otorisasi di sisi server, yang seharusnya menjadi lapisan utama dalam melindungi data pengguna.

Severity: High

Affected Endpoint

POST /newdaftar/get_all_data

Awal Penemuan

Pengujian dimulai saat melakukan eksplorasi terhadap fitur pendaftaran dan pengambilan data user. Setelah data berhasil disimpan, sistem menghasilkan sebuah ID unik yang digunakan untuk mengambil kembali data tersebut.

Pada tahap ini, muncul pertanyaan penting: apakah sistem benar-benar memverifikasi kepemilikan ID tersebut?

Untuk menjawab hal ini, dilakukan pengujian menggunakan intercept proxy untuk melihat bagaimana request dikirim ke server dan bagaimana sistem merespons.

Analisis Teknis

Request yang dikirim ke server memiliki parameter sederhana:

id=139116

Parameter ini digunakan oleh sistem untuk mengambil seluruh data user terkait. Namun, tidak ditemukan adanya validasi apakah ID tersebut milik user yang sedang login.

Ketika nilai ID diubah menjadi:

id=139117

Server tetap mengembalikan data, meskipun ID tersebut milik user lain. Hal ini mengindikasikan bahwa sistem sepenuhnya mempercayai input dari client.

Proof of Concept

Berikut demonstrasi eksploitasi kerentanan:

PoC: Data exposure via ID manipulation

Langkah Eksploitasi

  • Melakukan input data melalui form pendaftaran
  • Sistem menghasilkan ID unik (contoh: 139116)
  • Intercept request menggunakan proxy (Burp Suite)
  • Mengubah parameter ID ke milik user lain
  • Mengirim ulang request ke server
  • Server tetap mengembalikan data korban

Dampak

  • Akses tidak sah ke data pribadi (PII)
  • Kebocoran email dan nomor telepon
  • Eksposur data keluarga dan dokumen
  • Enumerasi massal data user (ID incremental)
  • Pelanggaran privasi yang serius

Akar Masalah

Masalah utama terletak pada tidak adanya kontrol otorisasi di sisi server. Sistem hanya menggunakan parameter ID tanpa melakukan verifikasi apakah user memiliki hak akses terhadap resource tersebut.

Selain itu, penggunaan ID incremental membuat eksploitasi menjadi jauh lebih mudah, karena attacker dapat melakukan enumerasi dengan hanya menambah atau mengurangi angka.

Insight & Pembelajaran

Dari kasus ini, terdapat beberapa hal penting yang dapat dipelajari:

  • ID bukanlah mekanisme keamanan
  • Authorization harus selalu dilakukan di server, bukan client
  • Data sensitif membutuhkan proteksi berlapis
  • ID incremental meningkatkan risiko eksploitasi massal

Temuan ini juga memperkuat bahwa banyak aplikasi masih mengabaikan prinsip dasar keamanan, terutama pada bagian kontrol akses.

Rekomendasi

  • Validasi bahwa ID milik user yang sedang login
  • Gunakan session-based authorization
  • Implementasi access control di setiap endpoint sensitif
  • Gunakan UUID untuk mencegah enumerasi
  • Tambahkan logging aktivitas mencurigakan

Kesimpulan

Kerentanan ini memungkinkan akses data pengguna lain tanpa izin hanya dengan memodifikasi parameter sederhana pada request.

Perbaikan segera sangat diperlukan untuk melindungi data pengguna dan menjaga integritas sistem dari potensi penyalahgunaan lebih lanjut.
