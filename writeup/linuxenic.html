Rate Limit Bypass - OTP Registration Abuse

Irsyad Bayu Kurniawan — Security Researcher

Write-up ini telah mendapatkan izin resmi untuk dipublikasikan oleh pihak LINUXENIC Corporation (responsible disclosure approved).

Overview

Write-up ini membahas temuan kerentanan pada fitur registrasi di platform LINUXENIC Corporation. Celah yang ditemukan tergolong sederhana, namun tetap memiliki dampak nyata jika dieksploitasi, terutama untuk spam dan abuse OTP.

Seluruh pengujian dilakukan dalam konteks responsible disclosure, dan kerentanan telah diperbaiki sebelum publikasi write-up ini.

Apa itu LINUXENIC?

LINUXENIC Corporation adalah platform Learning Management System (LMS) cybersecurity asal Indonesia yang berfokus pada pembelajaran hands-on melalui lab interaktif berbasis browser.

Pengguna dapat langsung mempraktikkan materi seperti Red Team dan Blue Team tanpa perlu setup tools atau virtual machine.

Ringkasan Kerentanan

Ditemukan celah pada fitur registrasi (sign up) yang memungkinkan pengiriman OTP (One-Time Password) tanpa batas akibat tidak adanya mekanisme rate limiting.

Endpoint Terdampak

POST https://linuxenic.com/wp-admin/admin-ajax.php
action=linuxenic_register

Deskripsi

Sistem pengiriman OTP tidak memiliki pembatasan request. Endpoint dapat diakses berulang kali tanpa:

  • CAPTCHA
  • Cooldown antar request
  • Rate limiting berdasarkan IP atau email

Akibatnya OTP dapat dikirim terus-menerus ke email target tanpa hambatan.

Langkah Reproduksi

1. Kirim POST request ke endpoint admin-ajax.php
2. Gunakan parameter valid (email, password, nonce)
3. Ulangi request beberapa kali (10–20x)
4. Server tetap mengirim OTP tanpa batas

Proof of Concept

for i in {1..10}; do
  curl -X POST "https://linuxenic.com/wp-admin/admin-ajax.php" \
  -F "email=target@email.com" \
  -F "action=linuxenic_register"
done

Hasil: OTP terkirim berulang kali tanpa batasan, tidak ada rate limit atau mekanisme proteksi tambahan.

Dampak

  • Email flooding ke target
  • Potensi abuse otomatisasi OTP
  • Brute force OTP karena tidak ada kontrol frekuensi

Rekomendasi

  • Implementasi rate limiting per IP/email
  • Tambahkan CAPTCHA (reCAPTCHA / hCaptcha)
  • Cooldown antar request
  • Monitoring aktivitas mencurigakan

Referensi

  • CWE-307: Improper Restriction of Excessive Authentication Attempts
  • CWE-799: Improper Control of Interaction Frequency
